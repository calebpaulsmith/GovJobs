JobList sticky toolbar — variants

You picked "sticky toolbar with search box + sort dropdown + multi-select facets". Three small variants below — same building blocks, different density. Pick one and I'll wire it to the live data.

Variant A · comfortable (default in the desktop mock)
47 of 312 matching current filters ↑ 23% above the trailing-90-day average for IL postings

What lives here

  • Search — in-list filter only; never adds chips to the global filter state.
  • Sort — pure ordering; saved with the view.
  • Facets — toggle-only narrowing of the visible list. Counts always reflect the global filter chips above.
  • Inline annotation — historical context (placeholder until the slice lands).

What does NOT live here

  • Agency / series / grade picks — those are global filter chips in the FilterPanel.
  • Geography — that's chips from polygon click + AddressSearch.
  • Saved-search management — that's the masthead pill.
Variant B · compact (facets behind a popover)
47 of 312 · GS, Remote ↑ 23% above 90d avg

Tradeoffs

  • Two lines instead of three — more room for rows.
  • One extra click to set facets, but power users with saved views are already past the picker.
  • Active facets show as a count badge on the button + comma-list in the summary.

Best for

  • Small screens (tablet) without going full mobile-dock.
  • Users who already know their facets and want to maximize row visibility.

Tradeoffs

  • Single source of truth for filter-in-list state — easy to encode in a URL.
  • Discoverability is the worry. The placeholder shows the syntax but new users won't read it.
  • Could degrade to Variant A if the user types plain prose without operators.

Best for

  • Power users + share-link consumers who land in a fully-configured view.
  • Keyboard-first navigation.

What gets saved in the share link / saved search (v2)

Toolbar state round-trips through the URL exactly like the global filters. Schema diff vs. v1:

{
  // v1 keys (unchanged)
  "filters":   { ... global chips ... },
  "metric":    "postings",
  "viewport":  { "center": [-87.65, 41.85], "zoom": 8 },
  "addressTarget": null,

  // v2 additions (D.5.28 / D.5.29)
  "viewMode":  "browse",
  "list": {
    "search":      "emergency",        // in-list search box value
    "sort":        "closing_soon",     // sort key
    "facets":      ["gs", "remote"],   // multi-select facet ids
    "selectedJobId": "FEMA-12345",     // currently-open row (D.5.29)
    "scrollOffset": 0.42               // fraction of list scrolled
  },
  "geographies": [
    { "type": "radius", "center": [-87.65, 41.85], "miles": 50, "label": "Chicago, IL" }
  ],
  "theme":     "dark"
}

The dock activeTab on mobile is intentionally NOT shared — it's a viewport-local preference, not part of the view itself. Opening a shared mosaic link on a phone resolves to mobile dock with the List tab active.
