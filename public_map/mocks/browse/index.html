Browse UI mocks

Static, click-through mocks for the Browse view (ADR-0033 / D.5.28) plus the iOS-style mobile dock that's been added on top. None of these are wired to real data — they're for picking a direction so we can lock in layout.ts, the JobList rich-mode shape, and where Pulse metrics live.

1. Desktop mosaic Default

Three panes per ADR-0033: small map top-left, SmallestAreaCard with a Pulse band top-right, JobList rich-mode with sticky toolbar bottom. Per-job historical badges live on individual rows.
Map List Area Metrics Saved

2. Mobile / web-app dock

Same content, single-pane. Five tabs in a bottom dock — Map / List / Area / Metrics / Saved. Tap a tab to swap the content area. Click between tabs in the mock to feel the transitions.

3. Metrics pane

What the "Metrics" tab (mobile dock) or the desktop Metrics drawer contains: trailing-12-month posting volume, agency mix, fill-rate, geographic concentration. Includes placeholders for the on-demand "AI call for this area" enrichment slot.

4. JobList sticky toolbar (zoom-in)

The toolbar above the rich rows: in-list search, sort dropdown, multi-select facet chips, inline historical annotation. This is the most-iterated component, so it gets its own dedicated mock.
How I'm reading your direction:

Mosaic-on-desktop / dock-on-mobile (your pick) means layout.ts needs a mode param (already in the ADR) and a third 'mobile-dock' breakpoint variant. Every mock here uses the existing CSS tokens from +layout.svelte so nothing changes visually for users who jump between Map-only and Browse. The sticky toolbar pattern (search + sort + facets) repeats inside the dock too, so the JobList feels identical across viewports.

Placeholder metric panels are clearly tagged with PLACEHOLDER badges so the data slice work can land separately without blocking the layout.
Existing dark-theme tokens (no new colors) Pill chips match FilterPanel / SavedSearchMenu Tooltip ⓘ pattern from D.5.16 InfoTooltip Light mode falls out of the tokens automatically
